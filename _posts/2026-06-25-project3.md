## 개요

태그 기반 파일 관리자 앱을 개발하면서 기기 내 모든 파일을 탐색하고 데이터베이스와 동기화하는 파일 스캐너를 구현했습니다.

개발 초기에는 에뮬레이터 환경에서 테스트를 진행했기 때문에 특별한 문제를 발견하지 못했습니다.
하지만 실제 이미지와 영상 파일이 약 300GB 저장된 기기에서 테스트를 진행하자 예상하지 못한 문제가 발생했습니다.

에뮬레이터에서는 정상적으로 완료되던 스캔이 실기기에서는 완료조차 되지 않았습니다.

이번 글에서는 문제를 분석하고, 직접 벤치마크를 설계해 최적 구조를 찾아낸 과정을 정리해보려고 합니다.

---

## 초기 구현

초기 구현은 다음과 같은 구조였습니다.

- **탐색 방식**: BFS (Queue 기반)
- **해시 계산**: MD5 전체 해시
- **디스패처**: `Dispatchers.IO.limitedParallelism(4)`

파일 탐색, 메타데이터 조회, DB 비교, 해시 계산 등 모든 작업이 동일한 Dispatcher 위에서 수행되고 있었습니다.

에뮬레이터에서는 문제가 없었습니다.

---

## 문제 발견

약 300GB 규모의 이미지·영상 데이터가 저장된 실기기에서 테스트를 진행하자 스캔이 완료되지 않는 문제가 발생했습니다.

로그를 분석한 결과 OOM(Out of Memory)으로 프로세스가 강제 종료되고 있었습니다.

원인을 분석해보니 두 가지가 결합된 문제였습니다.

**MD5 전체 해시**: 파일 크기만큼 메모리에 올려야 하므로, 수백 MB 파일을 여러 개 동시에 처리하면 힙이 급격히 증가합니다.

**BFS 탐색**: 같은 레벨의 디렉터리를 한꺼번에 큐에 적재하므로, 파일 수가 많을수록 동시 처리 대상이 폭발적으로 늘어납니다.

두 가지가 결합되면 `파일 수 × 파일 크기`만큼 힙이 증가하며 OOM이 발생합니다.

에뮬레이터에서 이 문제가 드러나지 않은 이유는 테스트 파일 수와 크기가 실제 환경과 달랐기 때문입니다.

---

## 1차 해결 - Partial Hash 도입

전체 파일을 읽는 대신 파일의 앞·중간·끝 일부만 읽는 Partial Hash 방식으로 변경했습니다.

```kotlin
// Before: 파일 전체를 읽어 MD5 계산
fun calculateFullHash(file: File): String {
    return file.readBytes().md5()
}

// After: 앞·중간·끝 일부만 읽어 계산
fun calculatePartialHash(file: File): String {
    // 파일의 특정 오프셋만 읽어 해시 계산
}
```

Partial Hash 도입 후 OOM 문제는 해결되었습니다.

한 가지 짚고 넘어갈 점이 있습니다.

당시에는 OOM 원인을 정확히 분리하지 못한 채 Partial Hash 도입, DFS 전환, Semaphore 적용을 동시에 진행했습니다. 이후 벤치마크를 통해 역추적한 결과, **OOM의 실질적 원인은 Partial Hash 도입으로 해결된 것**이었고 DFS 전환과 Semaphore는 OOM과 직접적인 관계가 없었습니다.

## 벤치마크 설계

OOM은 해결했지만 성능이 실제로 어떤 구조에서 가장 좋은지 데이터가 없었습니다.

탐색 방식(BFS/DFS)과 병렬 처리 수(`limitedParallelism(N)`)를 독립 변수로 분리해 직접 벤치마크를 수행했습니다.

### Cold Run vs Warm Run

벤치마크에서 중요한 개념입니다.

**Cold Run**: OS 페이지 캐시가 비어있는 상태에서의 측정입니다. 전원을 완전히 끄고 켠 뒤, 앱 데이터를 초기화(`pm clear`)한 상태에서 최초 1회 측정합니다. 실제 디스크 I/O 특성이 반영되므로 가장 현실적인 수치입니다.

**Warm Run**: 한 번 스캔이 완료된 이후 반복 측정하는 경우입니다. OS가 자주 접근한 데이터를 메모리에 캐싱하므로 실제 디스크 I/O가 줄어들어 빠르게 측정됩니다.

실험 결과 Warm Run은 조건이 달라도 62~70ms대로 수렴하여 조건 간 차이가 사라지는 것을 확인했습니다. 따라서 의미 있는 비교는 **Cold Run 기준**으로 수행했습니다.

### 측정 환경

- 실기기: 약 300GB 이미지·영상 데이터 저장
- 저장장치: 외장 SD카드 (`/storage/9C33-6BBD`)
- 측정 방식: 전원 OFF → ON → `adb shell pm clear` → 앱 실행 → 벤치마크 명령 실행
- 각 케이스 Cold Run 1회 측정

## 실험 결과

### limitedParallelism 비교 (BFS 고정)

| limitedParallelism | 탐색 방식 | Cold Run |
|---|---|---|
| 1 | BFS | 120,192ms |
| 2 | BFS | 124,239ms |
| 4 | BFS | **97,906ms** ← 최속 |
| 8 | BFS | 118,107ms |

### BFS vs DFS 비교 (limitedParallelism(4) 고정)

| limitedParallelism | 탐색 방식 | Cold Run |
|---|---|---|
| 4 | BFS | **97,906ms** ← 최속 |
| 4 | DFS | 145,089ms |

### Warm Run 참고

| limitedParallelism | 탐색 방식 | Warm Run (안정화 후) |
|---|---|---|
| 1 | BFS | ~65ms |
| 4 | BFS | ~63ms |
| 8 | BFS | ~68ms |
| 1 | DFS | ~67ms |
| 4 | DFS | ~65ms |

Warm Run은 조건이 달라도 수치가 수렴합니다. 조건 간 비교는 반드시 Cold Run 기준으로 해야 합니다.

## 예상과 달랐던 점

### Semaphore 가설이 틀렸다

초기에는 SD카드가 단일 버스 구조이므로 동시 디스크 접근을 Semaphore로 제한하면 성능이 개선될 것이라 예상했습니다.

```kotlin
private val semaphore = Semaphore(2)

val hash = if (hasPotentialRenameCandidate) {
    semaphore.withPermit {
        hashExtractor.calculatePartialHash(file)
    }
} else {
    existing?.fileHash
}
```

그러나 Semaphore 적용 버전과 미적용 버전을 비교한 결과 유의미한 성능 차이는 확인되지 않았습니다.

따라서 이번 테스트 환경에서는 Semaphore로 제한한 구간이 전체 성능에 큰 영향을 주는 병목은 아니었던 것으로 판단했습니다. 정확한 원인을 확인하려면 디스크 I/O 프로파일링 등 추가적인 분석이 필요합니다.

### BFS가 DFS보다 빨랐다

초기에 DFS로 전환한 이유는 메모리 안정화였는데, 성능 면에서는 BFS가 더 우수했습니다.

Cold Run 기준으로 BFS + limitedParallelism(4)는 97,906ms, DFS + limitedParallelism(4)는 145,089ms로 약 33% 차이가 났습니다.

다만 왜 BFS가 더 빨랐는지는 이번 실험만으로 설명할 수 없었습니다. 저장장치의 접근 패턴이나 운영체제의 캐시 정책 등이 영향을 미쳤을 가능성은 있지만, 이를 검증하지는 않았기 때문에 원인으로 단정하지 않겠습니다.

## 최종 구조 결정

벤치마크 결과를 바탕으로 최종 구조를 결정했습니다.

```kotlin
private val ioDispatcher = Dispatchers.IO.limitedParallelism(4)
private val traversalMode = TraversalMode.BFS

val hash = if (hasPotentialRenameCandidate) {
    hashExtractor.calculatePartialHash(file)  // Semaphore 제거
} else {
    existing?.fileHash
}
```

- **탐색 방식**: BFS
- **디스패처**: `Dispatchers.IO.limitedParallelism(4)`
- **Semaphore**: 제거 (유의미한 효과 없음)

Cold Run 기준 약 **97,906ms(약 98초)** 내 300GB 전체 파일 인덱싱 완료.

---

## 결론

이번 경험에서 얻은 교훈은 다음과 같습니다.

**가설은 틀릴 수 있다.** Semaphore가 SD카드 I/O를 제어한다는 가설, DFS가 성능상 유리하다는 가설 모두 실험에서 틀렸습니다. 가설을 고집하지 않고 측정 결과를 따라 구조를 바꿨습니다.

**변수를 분리해야 한다.** 초기에는 Partial Hash, DFS, Semaphore를 동시에 적용했기 때문에 어느 변경이 실제로 효과가 있었는지 알 수 없었습니다. 이후 독립 변수를 하나씩 바꾸며 측정한 결과 원인을 정확히 파악할 수 있었습니다.

**Cold Run과 Warm Run을 구분해야 한다.** Warm Run만 측정하면 조건이 달라도 수치가 수렴해 의미 있는 비교가 되지 않습니다. 실제 디스크 I/O 특성을 반영하려면 반드시 Cold Run 기준으로 측정해야 합니다.

**에뮬레이터는 실기기를 대체하지 않는다.** OOM도, I/O 병목도 에뮬레이터에서는 재현되지 않았습니다. 성능 관련 검증은 실제 사용자 환경에서 수행해야 합니다.
