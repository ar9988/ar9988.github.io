## 개요
태그 기반 파일 관리자 앱을 개발하면서 기기 내 모든 파일을 탐색하고 데이터베이스와 동기화하는 파일 스캐너를 구현했습니다.
개발 초기에는 에뮬레이터 환경에서 테스트를 진행했기 때문에 특별한 문제를 발견하지 못했습니다.
하지만 실제 이미지와 영상 파일이 약 300GB 저장된 기기에서 테스트를 진행하자 예상하지 못한 문제가 발생했습니다.
에뮬레이터에서는 정상적으로 완료되던 스캔이 실기기에서는 매우 오랜 시간이 소요되거나 사실상 완료되지 않는 수준의 성능 저하를 보였습니다.
처음에는 단순히 파일 수가 많아서 발생한 문제라고 생각했지만, 분석 결과 실제 병목은 CPU가 아닌 디스크 I/O에 있었습니다.
이번 글에서는 문제를 분석하고 최적화했던 과정을 정리해보려고 합니다.

## 초기 구현
파일 해시 계산은 비용이 큰 작업이므로 동시 실행 개수를 제한했습니다.
### dispatcher.limitedParallelism(4)
당시에는
* 파일 탐색
* 메타데이터 조회
* 데이터베이스 비교
* 문자열 정규화
* 해시 계산

⠀모든 작업이 동일한 Dispatcher 위에서 수행되고 있었습니다.
처음에는 문제가 없어 보였습니다.

## 문제 발견
실제 이미지 및 영상 파일이 약 300GB 저장된 기기에서 테스트를 진행하자 문제가 발생했습니다.
특히 외장 SD카드에 저장된 파일 비중이 높을수록 성능 저하가 심해졌습니다.
로그를 분석해보니 대부분의 시간이 파일 해시 계산 과정에서 소비되고 있었습니다.
파일 탐색이나 데이터베이스 조회는 빠르게 진행되었지만, 디스크에서 파일을 읽기 시작하는 순간 전체 처리 속도가 급격히 감소했습니다.
흥미로웠던 점은 에뮬레이터에서는 이러한 현상이 전혀 발견되지 않았다는 것입니다.
에뮬레이터 환경은 실제 저장장치의 특성을 완전히 재현하지 않기 때문에 병렬 I/O로 인한 병목이 드러나지 않았고, 결국 이 문제는 실기기 테스트를 통해서만 발견할 수 있었습니다.

## 왜 병렬성이 높을수록 느려질까?
처음에는 병렬성을 더 높이면 성능이 개선될 것이라고 생각했습니다.
하지만 저장장치는 CPU와 다르게 동작합니다.
특히 SD카드는 여러 파일을 동시에 읽기 시작하면 랜덤 접근 비용이 증가하며 저장장치 컨트롤러의 대기열이 커질 수 있습니다.
### 파일 A 읽기
### 파일 B 읽기
### 파일 C 읽기
### 파일 D 읽기
겉으로 보기에는 병렬 처리이지만 실제로는 서로가 디스크 접근을 경쟁하게 됩니다.
결과적으로 처리량은 증가하지 않고 오히려 전체 작업 시간이 늘어날 수 있습니다.

## 해결 방법
생각을 바꿨습니다.
모든 작업을 제한하는 것이 아니라 실제 병목이 발생하는 작업만 제한하기로 했습니다.

### semaphore.withPermit {

###  ├─ 병렬 수행

### 해시 계산

## 왜 Semaphore(2) 였을까?
실제 병목은 CPU 연산이 아니라 디스크에서 파일을 읽는 과정이었다.

따라서 중요한 것은 최대 병렬성이 아니라 디스크 I/O에 과도한 부하를 주지 않는 것이었다.

Semaphore 개수는 다양한 문헌과 경험을 참고하여 우선 2로 설정하였고, 실제 테스트 환경에서 기존 구현 대비 충분한 성능 향상을 확인할 수 있었다.

다만 1, 2, 4, 8 등 다양한 값에 대한 체계적인 벤치마크는 수행하지 않았기 때문에, Semaphore(2)가 절대적인 최적값이라고 단정할 수는 없다.

향후 저장장치 종류 및 파일 수에 따른 벤치마크를 진행해 볼 예정이다.

## Semaphore가 좋았던 이유
## 1\. 가벼운 작업은 그대로 병렬 수행
폴더 탐색
메타데이터 조회
DB 비교
문자열 정규화
등의 작업은 계속 병렬 수행됩니다.
실제 병목이 발생하지 않는 작업까지 제한할 필요는 없었습니다.

## 2\. 디스크 I/O만 선택적으로 제한
실제 병목이 발생하는 파일 읽기 작업만 제한했습니다.
덕분에 디스크는 과도한 랜덤 접근 없이 안정적으로 파일을 읽을 수 있었고 전체 처리량도 개선되었습니다.

## 3\. 스레드를 점유하지 않는다
가장 흥미로웠던 점은 Semaphore.withPermit() 의 동작 방식이었습니다.
처음에는 단순히 동시 실행 개수를 제한하는 도구라고 생각했습니다.
하지만 코루틴용 Semaphore는 대기 중인 작업이 스레드를 점유하지 않습니다.
차례가 올 때까지 코루틴을 Suspend 상태로 전환하고 스레드는 다른 작업에 반환합니다.
덕분에 디스크 I/O를 기다리는 동안에도 다른 가벼운 작업들이 계속 수행될 수 있었습니다.

## 추가로 알게 된 점
흥미로운 사실 하나를 발견했습니다.
일부 라이브러리는 내부적으로 이미 Dispatchers.IO를 사용하도록 구현되어 있습니다.
이 경우 상위 계층에서 Main 스레드에서 호출하더라도 내부에서 자동으로 I/O 스레드로 전환됩니다.
다만 저는 다음과 같은 이유로 상위 계층에서도 명시적으로 Dispatchers.IO를 사용하는 편입니다.
* 코드만 보고도 무거운 작업임을 알 수 있음
* 유지보수 시 실수를 줄일 수 있음
* 내부 구현이 변경되더라도 안전함

⠀성능 때문이라기보다는 코드 의도를 명확하게 표현하기 위한 선택에 가깝습니다.

## 정리
이번 경험을 통해 얻은 가장 큰 교훈은 다음과 같습니다.
성능 문제는 더 많은 스레드를 사용하는 것으로 해결되지 않는다.
중요한 것은 병목이 발생하는 지점을 정확하게 찾는 것이었습니다.
무작정 병렬성을 높이는 대신 실제 병목 구간만 선택적으로 제어했을 때 훨씬 좋은 결과를 얻을 수 있었습니다.
또한 에뮬레이터에서 문제가 없다고 해서 실제 환경에서도 문제가 없다는 의미는 아니라는 사실을 다시 한번 체감할 수 있었습니다.
성능 최적화는 결국 실제 사용자 환경에서 검증되어야 한다는 점을 배울 수 있었던 경험이었습니다.
